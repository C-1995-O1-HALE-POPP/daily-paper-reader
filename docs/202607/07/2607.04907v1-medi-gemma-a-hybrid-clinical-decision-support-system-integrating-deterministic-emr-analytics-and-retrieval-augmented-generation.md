## 摘要
在高风险临床环境中部署大型语言模型（LLM）仍然受到结构性幻觉、对表格患者数据的弱确定性推理以及向量检索中的遗漏的限制。本文介绍了Medi-Gemma的架构和验证，这是一个用于伤口病理分诊和工作流自动化的临床决策支持系统（CDSS）。该平台引入了一种解耦框架，将临床感知与数据编排分离，同时保留可追溯推理。Medi-Gemma使用由集中式ClinicalOrchestrator协调的多阶段流水线。数据请求由DataManager在没有生成式推理的情况下处理，该管理器通过类型强制清理非结构化电子病历（EMR）文件。自然语言查询由分层IntentRouter处理，该路由器将请求路由到由PandasQueryEngine执行的确定性分析路径，或由使用CPU优化向量存储的ClinicalRAGEngine管理的患者特定推理。一个关键贡献是Ground Truth Injection Module，它拦截患者特定查询，提取数字识别令牌，通过Pandas查询结构化数据框，检索最新的已验证临床状态，并在生成前将此快照作为覆盖上下文块嵌入LLM提示中。安全性合规由确定性ProtocolManager强制执行，该管理器将临床术语映射到固定的循证风险路径，而SafetyVerifier短语过滤器防止输出规则违反。验证表明，该架构消除了语义上下文漂移，防止了数据库编译崩溃，并提高了对后端临床存储库的事实遵循。这些结果支持Medi-Gemma作为基于LLM的临床决策支持的一种更安全的模式，其中结构化数据保真度、检索基础和确定性保障至关重要。

## Abstract
Deploying Large Language Models (LLMs) in high-stakes clinical settings remains limited by structural hallucinations, weak deterministic reasoning over tabular patient data, and omissions in vector retrieval. This paper presents the architecture and validation of Medi-Gemma, a Clinical Decision Support System (CDSS) for wound pathology triage and workflow automation. The platform introduces a decoupled framework that separates clinical perception from data orchestration while preserving traceable reasoning. Medi-Gemma uses a multi-stage pipeline coordinated by a centralized ClinicalOrchestrator. Data requests are handled without generative inference by a DataManager that cleans unstructured Electronic Medical Record (EMR) files through type coercion. Natural language queries are processed by a hierarchical IntentRouter, which routes requests to deterministic analytics paths executed by a PandasQueryEngine or to patient-specific reasoning managed by a ClinicalRAGEngine using a CPU-optimized vector store. A key contribution is the Ground Truth Injection Module, which intercepts patient-specific queries, extracts numeric identification tokens, queries the structured dataframe via Pandas, retrieves the latest validated clinical state, and embeds this snapshot as an overriding context block in the LLM prompt before generation. Safety compliance is enforced by a deterministic ProtocolManager that maps clinical terminology to fixed evidence-based risk pathways, while a SafetyVerifier phrase filter prevents output rule violations. Validation shows that this architecture eliminates semantic context drift, prevents database compilation crashes, and improves factual adherence to backend clinical repositories. These results support Medi-Gemma as a safer pattern for LLM-based clinical decision support where structured data fidelity, retrieval grounding, and deterministic safeguards are essential.

## 1. 核心问题与研究动机（背景）

- **问题**：大型语言模型（LLM）在高风险临床场景中部署受限，主要因为：
  - **结构性幻觉**：在结构化表格数据（如患者生命体征、实验室指标）上无法进行精确的确定性和数值推理。
  - **向量检索遗漏**：传统检索增强生成（RAG）系统将文本随意分块，可能遗漏关键历史上下文或优先选用过时的病历记录。
- **目标**：构建一个混合式临床决策支持系统（CDSS），将确定性EMR分析与检索增强生成相结合，确保事实一致性、可追溯性和安全性。

## 2. 方法论

### 核心思想
- **解耦架构**：将临床感知（LLM推理）与数据编排（确定性计算）分离，通过集中式`ClinicalOrchestrator`协调多阶段流水线。
- **动态意图路由（IntentRouter）**：基于规则（正则匹配和关键词）将自然语言查询分为三类：
  - **患者特定查询**（包含患者ID或`patient`+数字）→ 路由到临床路径（`ClinicalRAGEngine`）。
  - **统计聚合查询**（如`average`、`count`、`percentage`）→ 路由到数据分析路径（`PandasQueryEngine`）。
  - **视觉/上下文跟踪查询**（例如“这个伤口”）→ 路由到临床路径，维持会话连续性。
- **Ground Truth State Injection（真实状态注入）**：关键创新。
  - 提取查询中的患者ID → 直接通过Pandas在结构化数据框（DataFrame）中执行精确查询 → 获取该患者最近一次经确认的临床状态（日期、伤口尺寸、组织成分等） → 将该快照作为**覆盖性上下文块**注入LLM提示中，并附带指令“你必须使用此日期作为当前状态”。
- **确定性安全层**：
  - `ProtocolManager`：根据预定义的`protocols.yaml`将临床术语映射到固定风险等级（如“关键感染”→“Critical Risk”），采用关键词匹配优先级。
  - `SafetyVerifier`：关键词过滤，替换违规输出（如未授权的剂量指令）为安全警报。

### 关键技术细节
- **数据预处理**：`DataManager`对非结构化EMR文件进行类型强制转换（如将`N/A`转为null），标准化字段名（如`patient_id`），填充缺失数值为0.0，保留文本缺失为占位符。
- **向量检索**：使用本地CPU部署的`BAAI/bge-small-en-v1.5`模型，通过LlamaIndex实现，并利用`MetadataFilters`（按患者ID精确匹配）防止跨患者上下文污染。
- **推理引擎**：后端使用Gemma (via Ollama)作为LLM；数据分析引擎直接生成Python代码（如`.mean()`）并通过Pandas执行。

### 算法流程（文字说明）
1.  医生上传结构化和非结构化EMR数据集。
2.  输入自然语言查询。
3.  `IntentRouter`分类查询：
    - 若为聚合查询 → 直接调用`PandasQueryEngine`生成并执行代码 → 返回精确数值结果。
    - 若为患者查询 → 调用`ClinicalOrchestrator`：
        - 提取患者ID → 在DataFrame中精确查找最新记录 → 构建真实状态块 → 注入LLM提示。
        - 可选：`ClinicalRAGEngine`检索该患者补充叙事记录。
        - LLM生成响应。
4.  `ProtocolManager`评估生成内容，映射风险等级。
5.  `SafetyVerifier`过滤违规短语。
6.  返回最终响应给临床医生。
