## 摘要
我们提出了PhysicianBench，一个用于评估LLM智能体在电子病历（EHR）环境中基于真实临床场景的医生任务的基准测试。现有的医学智能体基准测试主要关注静态知识回忆、单步原子动作或动作意图，缺乏对环境可验证执行的评估。因此，它们无法捕捉真实临床系统中长期、复合的工作流程。PhysicianBench包含100个从基层医疗和专科医生之间的真实会诊案例改编的长期任务，每个任务由独立的医生小组审核。任务在具有真实患者记录的EHR环境中实例化，并通过商业EHR供应商使用的相同标准API访问。任务涵盖21个专业（如心脏病学、内分泌学、肿瘤学、精神病学）和多样的工作流程类型（如诊断解读、药物处方、治疗计划），每个任务平均需要27次工具调用。解决每个任务需要跨就诊检索数据，推理异构临床信息，执行重要的临床操作，并生成临床文档。每个任务被分解为结构化的检查点（整个基准测试中共670个），这些检查点捕捉不同的完成阶段，并由任务特定脚本进行执行验证评估。在13个专有和开源LLM智能体中，表现最佳的模型仅达到46%的成功率（pass@1），而开源模型最高达到19%，揭示了当前智能体能力与现实临床工作流程需求之间的巨大差距。PhysicianBench为衡量自主临床智能体的进展提供了一个真实且基于执行验证的基准测试。

## Abstract
We introduce PhysicianBench, a benchmark for evaluating LLM agents on physician tasks grounded in real clinical setting within electronic health record (EHR) environments. Existing medical agent benchmarks primarily focus on static knowledge recall, single-step atomic actions, or action intent without verifiable execution against the environment. As a result, they fail to capture the long-horizon, composite workflows that characterize real clinical systems. PhysicianBench comprises 100 long-horizon tasks adapted from real consultation cases between primary care and subspecialty physicians, with each task independently reviewed by a separate panel of physicians. Tasks are instantiated in an EHR environment with real patient records and accessed through the same standard APIs used by commercial EHR vendors. Tasks span 21 specialties (e.g., cardiology, endocrinology, oncology, psychiatry) and diverse workflow types (e.g., diagnosis interpretation, medication prescribing, treatment planning), requiring an average of 27 tool calls per task. Solving each task requires retrieving data across encounters, reasoning over heterogeneous clinical information, executing consequential clinical actions, and producing clinical documentation. Each task is decomposed into structured checkpoints (670 in total across the benchmark) capturing distinct stages of completion graded by task-specific scripts with execution-grounded verification. Across 13 proprietary and open-source LLM agents, the best-performing model achieves only 46% success rate (pass@1), while open-source models reach at most 19%, revealing a substantial gap between current agent capabilities and the demands of real-world clinical workflows. PhysicianBench provides a realistic and execution-grounded benchmark for measuring progress toward autonomous clinical agents.

## 1. 核心问题与整体含义（研究动机和背景）
- **现有局限性**：当前医疗领域的大语言模型（LLM）基准测试主要聚焦于**静态知识回忆**（如 MedQA、PubMedQA）、**单步原子动作**（如 MedAgentBench 的单步检索或下单），或**动作意图**而不验证实际系统状态。这些基准无法评估 LLM 在**真实电子病历（EHR）环境**中完成**长时、复合工作流**的能力——即需要跨就诊检索数据、推理异构信息、执行关键操作（如开药、转诊、生成文书）并最终产生可验证结果的场景。
- **现实需求**：临床医生工作负担沉重，若 LLM 智能体能可靠地执行 EHR 操作，可显著缓解职业倦怠。但缺乏合适的评估工具来量化进展。
- **研究目标**：构建一个**真实、可执行验证**的基准，测试 LLM 智能体在真实 EHR 环境（基于 HL7 FHIR 标准）中完成长时临床任务的能力，填补现有基准的空白。

## 2. 论文提出的方法论
### 核心思想

- 每个任务包含自然语言指令 \( I \)（定义角色、触发事件、步骤、交付物）和一组评估检查点 \( C = \{c_1,...,c_K\} \)。

2. **EHR 环境**：基于 **HAPI FHIR JPA 服务器**（FHIR R4 标准）搭建，每个任务在独立 Docker 容器中运行，预加载对应患者的标准化数据（LOINC 编码）。
3. **工具接口**：14 种工具，包括搜索（患者、诊断、化验、用药、生命体征、临床笔记等）和创建（药物订单、服务请求、预约、通信），以及文件写入。所有工具使用函数调用（function-calling）格式。
4. **检查点评估**：每个任务分解为 **670 个检查点**（总跨 100 个任务），分为四类：
   - **数据检索**：验证是否查询了必要资源
   - **临床推理**：细分为计算分数/解读/判断/应用标准
   - **动作执行**：验证是否创建了正确的 FHIR 资源（药物、检查订单等）

5. **验证流程**：11 名临床医生通过**多轮人机交互管线**审核任务（初始审核→智能体修改→差异审批→迭代至通过）。

## 3. 实验设计
### 数据集/场景
- **PhysicianBench 基准**：100 个长时任务，来自真实会诊案例，覆盖 **21 个专科**（分组为 8 大类别：心脏、内分泌、消化、免疫/感染、精神/神经/成瘾、血液/肿瘤、肾/泌尿、肺/其他）。
- 任务类型分布：诊断与解读 13%、药物处方 26%、治疗计划 27%、检查与风险分层 34%。
- 平均每个任务需要 **27 次工具调用**，最长可达 100 步。

### 对比的方法（模型）
- **专有模型（10 个）**：GPT-5.5、GPT-5.4、Claude Opus 4.6、Claude Opus 4.7、Claude Sonnet 4.6、MiMo-v2.5-Pro、Qwen3.6-Plus、MiniMax M2.7、Gemini Pro 3.1、Grok-4.20。
- **开源模型（2 个）**：DeepSeek V4-Pro、Kimi-K2.6。
- 所有模型使用同一轻量工具调用框架（无第三方 SDK），最大 100 步，温度默认，支持并行工具调用。

### 评估指标
- **Pass@1**：单次完成率（3 次运行平均）。
- **Pass@3**：至少一次成功的概率（unbiased estimator）。
- **Pass^3**：三次都成功的概率（可靠性）。
- 平均工具调用次数（#Turns）。

## 6. 主要结论与发现
- **整体表现不佳**：最佳模型 GPT-5.5 pass@1 仅 **46.3%**，GPT-5.4 27.7%，Claude Opus 4.6 31.7%。最强开源模型 DeepSeek V4-Pro 18.7%。
- **可靠性低**：即使最佳模型，三次运行都成功的概率（Pass^3）仅 **28.0%**，多数模型低于 10%，说明任务完成具有很大随机性。
- **错误主要集中在临床推理**：占全部失败检查点的 **50.4%**（平均），数据检索占 12.4%，动作执行 18.8%，文档 18.4%。说明瓶颈是**多步临床解释**而非工具使用。
- **GPT-5.5 领先原因**：不是临床知识更强，而是**更一致的数据引用**（避免编造事实）和**更全面的检查清单覆盖**，且使用了更多工具调用（41.9 次 vs Claude Opus 4.6 的 25.2 次）。
- **专科差异大**：内分泌表现最好（GPT-5.5 59.0%），精神/神经/成瘾最差（33.3%），不同模型在不同专科各有优劣。
- **任务类型难度**：治疗计划最难（最佳 40.7%），检查与风险分层相对容易（54.9%）。

## 7. 优点
- **高临床真实性**：任务源自真实会诊案例，经独立医生多轮审核，患者数据来自脱敏真实 EHR 并保留共病模式。
- **执行可验证**：智能体动作通过查询 FHIR 服务器验证，而非仅检查意图或自然语言输出，杜绝“说而不做”的幻觉。
- **细粒度评估**：670 个检查点按类型分类，允许诊断失败环节（检索/推理/动作/文档），提供可操作的改进方向。
- **标准环境**：使用 HL7 FHIR 国际标准，与商业 EHR 系统接口一致，结果可复现、可推广。
- **开源全栈**：任务、环境、框架、评估工具全部开源，支持公平比较和未来研究。
- **涵盖多样性与规模**：21 个专科、4 大类任务、100 个任务，平均 27 步，复杂度远超此前基准。

## 8. 不足与局限
- **工作流覆盖有限**：仅针对门诊会诊场景，未包含住院（如出院小结、用药管理）、急诊、多科室协作等更复杂的临床工作流。
- **缺乏多模态**：当前仅处理结构化数据和文本笔记，未纳入影像、波形、基因数据等，医生实际工作中常使用这些信息。
- **未评估协作能力**：任务设计为自主完成，未测试智能体与人类医生或患者的交互、共享决策、或面对临床约束（如资源限制、患者偏好）时的表现。
- **未评估安全性**：虽然检查点包含安全相关要求（如避免危险推荐），但缺乏系统性安全评估（如假阳性/假阴性率、对抗鲁棒性）。
