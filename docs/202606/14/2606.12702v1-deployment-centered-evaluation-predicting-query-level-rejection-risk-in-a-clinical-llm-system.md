## 摘要
大语言模型（LLM）正日益融入临床系统，因此评估这些系统的真实世界效用至关重要。然而，静态基准测试倾向于衡量正确性而非用户接受度，跨查询聚合性能，且需要密集标注的数据集——这导致评估临床系统时存在重大盲点。在本工作中，我们对嵌入某学术医疗中心电子健康记录中的LLM系统进行了以部署为中心的评估，其中用户反馈稀疏但密切反映部署条件。具体而言，我们训练了一个响应前分类器，基于查询内容和生成前可用的部署特定上下文，估计未来交互导致用户拒绝LLM响应的风险。我们基于4.5个月的用户反馈对我们的模型进行了前瞻性分析，发现预测模型的AUROC达到0.719。进一步，我们估计了此类预测在两个下游用例（护栏触发和弃权）中的益处。我们的关键概念洞察是：利用部署特定上下文（即提供者类型、科室名称、用于响应的语言模型），而非仅查询内容，能提高预测用户是否拒绝系统输出的能力。总体而言，我们的实证案例研究证明了利用部署特定上下文预测用户拒绝的可行性，为定向护栏开辟了道路。

## Abstract
Large language models (LLMs) are increasingly integrated into clinical systems, making it essential to evaluate the real-world utility of these systems. However, static benchmarks tend to measure correctness rather than user acceptance, aggregate performance across queries, and require densely annotated datasets -- leading to major blind spots for evaluating clinical systems. In this work, we perform a deployment-centered evaluation of an LLM system embedded within electronic health records at an academic medical center, where user feedback is sparse but closely reflects the deployment conditions. Specifically, we train a pre-response classifier that estimates the risk that a future interaction will result in the user rejecting the LLM response, based on query content and deployment-specific context available before generation. We conduct a prospective analysis of our model over 4.5 months of user feedback, finding that our prediction model achieves an AUROC of 0.719. Further, we estimate the benefit of such predictions in two downstream use cases (guardrail triggering and abstention). Our key conceptual insight is that making use of deployment-specific context (i.e., the provider type, department name, language model used for response), as opposed to only query content, improves the ability to predict whether the user will reject the system output. Altogether, our empirical case study demonstrates the feasibility of predicting user rejection using deployment-specific context, opening the door to targeted guardrails.

## 2. 方法论

- **核心思想**：训练一个**响应前预测模型**，利用**查询内容+部署特定上下文**（提供者类型、科室名称、语言模型、上下文长度）估计用户拒绝LLM响应的概率。模型在用户提交查询后、LLM生成响应前运行，从而允许低延迟、低成本地触发干预（如弃权或护栏激活）。
- **关键技术细节**：
  - **任务形式化**：给定查询 \( q_i \) 及关联的部署上下文（提供者类型 \( p_i \)、科室 \( d_i \)、模型 \( m_i \)、上下文长度 \( l_i \)），学习评分函数 \( f(q_i, p_i, d_i, m_i, l_i) \) 预测二元拒绝标签 \( r_i \in \{1 (拒绝), -1 (接受)\} \)。
  - **模型结构**：采用**逻辑回归**（3000次最大迭代，平衡类别权重）。查询 \( q_i \) 使用 `text-embedding-3-large` 转为嵌入向量；分类特征（提供者类型17维、科室129维、模型2维）进行独热编码。
  - **特征选择**：遍历所有可能的31种特征子集，在验证集上（4周数据）选择最大化AUROC的最佳组合。最终选定**嵌入 + 提供者类型 + 科室 + 模型**。
  - **训练流程**：初始训练集用前8周带反馈的数据，验证集用接下来4周，测试期为后续19周。每周测试后，将该周真实标签数据加入训练集**动态重新训练**模型（保持特征集和超参数不变），以适应分布漂移。
  - **超参数优化**：对SAGA求解器、L2和弹性网络惩罚系数进行网格搜索（C ∈ {0.001, 0.005, …, 1}，l1_ratio ∈ {0.05, 0.5, 0.7, 0.9, 1.0}）。

## 3. 实验设计

- **数据来源**：某学术医疗中心EHR内嵌的LLM系统日志，共74,729次交互（2025/09/01–2026/04/11）。其中1,196次有明确二元反馈（👍/👎）。经过去重、筛选仅保留第一次提问、去除缺失关键字段的红队测试用例后，最终得到**878条带注释的查询-标注对**（标注比例约1.2%）。
  - 初始训练集：前8周（09/01–10/30，283条）
  - 验证集：随后4周（10/30–11/27，162条）
  - 测试集：后续19周（11/27–04/11，剩余数据）
- **基准方法**：
  - **仅嵌入**：使用相同逻辑回归，但特征仅为查询嵌入。
  - **单一上下文特征**：分别用部门、提供者、模型、上下文长度单独训练。

  - **随机基线**：AUROC = 0.5。
  - **始终接受/始终拒绝**：用于F1比较。
- **评估指标**：
  - **AUROC**（主要指标）：衡量全阈值下的区分能力。使用Bootstrap（2000次迭代）计算95%置信区间。

  - **Fβ**：用于下游用例（不同β权衡精确率与召回率）。

## 7. 优点

- **真实部署验证**：使用实际临床用户产生的稀疏反馈数据，避免了静态基准的三大盲点（正确性、聚合性、密集标注）。
- **前瞻性评估**：采用滚动窗口测试，符合实际部署中模型持续更新的场景，比历史切分更可靠。
- **轻量级模型**：逻辑回归仅需CPU单节点训练，计算成本极低，易于部署和重训练。
- **部署上下文的价值洞察**：系统揭示了“谁在问、从哪问、用什么模型问”比“问了什么”更重要，为临床AI系统设计提供了新视角。
- **双用途阈值调节**：同一个模型通过调整β可服务于不同安全需求（弃权vs.护栏），实用性高。
- **定性分析补充**：结合具体用户反馈示例，直观展示了提供者类型和科室如何影响拒绝行为。

## 8. 不足与局限

- **数据稀疏性与选择偏差**：仅1.6%的交互有反馈，且不同科室/提供者的反馈率不均（如住院医师反馈少，注册护士反馈多），可能导致模型偏向“更愿意反馈”的用户群体。作者指出需要未来工作分析这一偏差。
- **单一机构限制**：所有数据来自同一学术医疗中心的特定EHR配置、患者群体和AI工具实践，结论可能不直接推广到不同环境。
- **模型简单**：仅使用逻辑回归，未尝试更复杂的深度模型（可能提升嵌入特征利用效率），但这也被作者视为一种受数据稀疏性限制的权衡。
- **未利用未标注数据**：大量无反馈的交互被丢弃，未来的半监督或弱监督方法可能提升性能。
- **缺乏外部验证**：没有在另一数据集或时间段上验证模型泛化能力（除了每周的滚动测试）。
- **红队用例排除**：11条红队测试用例被排除，但红队交互可能代表重要的拒绝模式，纳入需谨慎处理。
- **反馈噪音**：用户点击“踩”可能出于多种原因（错误、格式不满意、个人偏好），模型无法区分本质原因。
